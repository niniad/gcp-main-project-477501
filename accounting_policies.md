# EC事業 会計方針
*最終更新: 2026-02-25*

本ドキュメントは、BQ会計システムで採用している会計処理の方針と根拠を記録する。

---

## 1. 仕訳日基準: settlement_end_date vs deposit_date

### 1.1 Amazon精算レポートの日付カラム

BQの `sp_api_external` データセットには以下の日付が存在する:

| カラム | 内容 | 例 |
|---|---|---|
| settlement_start_date | 精算期間の開始日 | 2025-12-16 |
| settlement_end_date | 精算期間の終了日 | 2025-12-29 |
| **deposit_date** | **Amazonが銀行に入金する日** | 2025-12-31 |
| posted_date | 個別トランザクションの計上日 | 各種 |

### 1.2 deposit_date と settlement_end_date の関係

BQデータの調査結果（2023年〜2026年の全件）:

> **deposit_date = settlement_end_date + 2日（常に一定）**

例:
- settlement_end_date: 2025-12-29 → deposit_date: 2025-12-31
- settlement_end_date: 2026-01-12 → deposit_date: 2026-01-14
- settlement_end_date: 2025-06-30 → deposit_date: 2025-07-02（月またぎ）

### 1.3 現行の設定

BQ `accounting.settlement_journal_view` は **deposit_date** を仕訳日として使用している（2026-02-25に移行完了）。

```
settlement_journal_view.booking_date = deposit_date  ★ 変更済み
  → settlement_journal_payload_view.issue_date = booking_date
    → journal_entries.journal_date = DATE(issue_date)
```

> 移行に伴い、年度またぎ精算（settlement 11898495253: settle=2024-12-30→deposit=2025-01-01）の
> P&L寄与 ¥62,501 が 2024→2025 に移動したため、差異調整仕訳 Id=193 で補填済み。

### 1.4 deposit_date を採用した理由

| 観点 | settlement_end_date | deposit_date（推奨） |
|---|---|---|
| 発生主義の正確性 | 精算期間の終了日 = やや正確 | 入金日 = 2日遅れ（実質無視可能） |
| 銀行残高との照合 | 一致しない → 消し込み必要 | **完全一致 → 自動照合可能** |
| freee連携 | 売掛金勘定の介在が必要 | **直接入金として登録可能** |
| 年またぎ影響 | 年末精算が年を跨ぐ可能性（低い） | 同上（2日差のため同様） |
| P&L精度 | ほぼ同等 | ほぼ同等（2日差） |

**実施済み（2026-02-25）:**
全期間を deposit_date に変更し、差異調整仕訳を再計算済み。
- 2023年: 年度またぎ精算なし → P&L変動なし
- 2024年: 1件の年度またぎ精算 → 差異調整仕訳 Id=193 で補填済み

### 1.5 freee連携時のフロー（deposit_dateベース）

```
【deposit_dateベース（推奨）】
Amazon精算 → deposit_date で仕訳登録
  Dr. 楽天銀行 / Cr. 売上高, Dr. 各手数料科目 / Cr. 楽天銀行
  → 銀行明細と直接照合可能（freee「自動で経理」機能が使える）

【settlement_end_dateベース（現行）】
Amazon精算 → settlement_end_date で仕訳登録
  ① Dr. 売掛金-Amazon / Cr. 売上高, Dr. 各手数料 / Cr. 売掛金-Amazon
  ② deposit_date に Dr. 楽天銀行 / Cr. 売掛金-Amazon（消し込み仕訳）
  → 2段階の処理が必要で複雑
```

## 2. 為替差損益の処理

### 2.1 現在の方針

為替差損益は独立した勘定科目として計上する（MFのように仕入高に内包しない）。

### 2.2 科目分類

| 設定項目 | 値 |
|---|---|
| account_name | 為替差損益 |
| small_category | **経費**（2026-02-25に変更済み。旧: 収入金額） |
| large_category | 損益 |

**表示:** P&Lの費用セクションに表示（経費の debit = 費用計上）

→ **P&L合計への影響はゼロ。** 収入金額→経費への変更は表示位置のみの変更。

> 注: 「雑費」に入れるのではなく、**「為替差損益」という科目名のまま** `small_category` を `経費` に変更した。
> 雑費は通常の事業経費用（事務用品等）であり、為替差損を混ぜると管理が不明瞭になる。

### 2.3 為替差損益の発生パターン

代行会社（ESPRIME/YP/THE直行便）への送金時に発生:

1. 日本から CNY 口座に振替（レートA）
2. 代行会社が CNY で支出（レートB）
3. A ≠ B の場合、JPY換算で差異が生じる → 為替差損益

**2023年の実績:**
| 取引 | 日付 | 金額（損失） |
|---|---|---|
| YP 為替差損 | 2023-06-21 | -¥24,376 |
| THE直行便 引き落とし差額 | 2023-09-07 | -¥14,725 |
| **合計** | | **-¥39,101** |

## 3. 期末為替評価替え

### 3.1 概要

代行会社への預け金は外貨（CNY）建てで管理されている。期末に CNY 残高がある場合、期末公示レートで評価替えし、帳簿上の JPY 残高との差額を「為替差損益」として計上すべき。

### 3.2 現在の状況

BQで確認した期末 CNY 残高と JPY 換算:

| 年末 | CNY残高 | 最終取引レート | 帳簿JPY残高 | 公示レート評価JPY | 差額 |
|---|---|---|---|---|---|
| 2022 | 3,978 CNY | 19.64 | ¥80,441 | ¥78,130 | **-¥2,311** |
| 2023 | 28,431 CNY | 20.63 | ¥586,560 | ¥586,560 | ¥0 |
| 2024 | 8,686 CNY | 23.10 | ¥200,657 | ¥200,657 | ¥0 |
| 2025 | 392 CNY | 22.36 | ¥8,759 | ¥8,759 | ¥0 |

> 現在の振替テーブルでは `balance_jpy` が最終取引レートで再計算されているため差額ゼロ。
> ただし本来は「期末の公示レート（税関レート等）× CNY残高」と「取引レートの積み上げ残高」の差を評価すべき。

### 3.3 期末評価替えの判断基準

- **CNY残高が小さい場合**（例: 2025年末 ¥8,759）: 為替変動の影響は軽微 → 評価替え省略可
- **CNY残高が大きい場合**（例: 2023年末 ¥586,560）: 為替レートの1円変動で ¥28,431 の影響 → 評価替え推奨

### 3.4 評価替えの仕訳方法

**記録場所: 手動仕訳テーブル**（推奨）

振替テーブルは代行会社との実取引（送金・支払い）を記録するもの。期末評価替えは取引ではなく会計上の調整であるため、手動仕訳テーブルに記録する方がクリーン。

**仕訳パターン:**
```
【CNY安 / JPY高 → 含み損（評価減）】
  Dr. 為替差損益 / Cr. 預け金（ESPRIME等）

【CNY高 / JPY安 → 含み益（評価増）】
  Dr. 預け金（ESPRIME等） / Cr. 為替差損益
```

**手動仕訳テーブルへの入力:**
| 項目 | 値 |
|---|---|
| 仕訳日 | 12月31日 |
| 借方科目 | 為替差損益（損の場合）/ 預け金（益の場合） |
| 貸方科目 | 預け金（損の場合）/ 為替差損益（益の場合） |
| 金額 | 差額の絶対値 |
| 摘要 | 「期末為替評価替え（ESPRIME CNY残高 XX元 × YY円/元 = ZZ円、帳簿残高 WW円、差額 VV円）」 |
| ソース | （空欄 or 新規追加: 「評価替え」） |
| 仕訳区分 | （空欄 or 新規追加: 「期末評価替え」） |

### 3.5 評価替えの翌期首処理

評価替え仕訳は期末のみの調整であり、翌期首に**洗替え（逆仕訳）を行うかどうか**は方針による:

- **洗替え方式**: 翌1/1に逆仕訳を入れ、翌期の為替差損益は取引ベースで計上
- **切放し方式**: 洗替えしない。翌期の帳簿残高は評価替え後の値

個人事業主の実務では切放し方式（洗替えしない）が簡便。

## 4. 開業費（繰延資産）

### 4.1 現在の状態

- NocoDB **開業費専用テーブル** に **103件** のエントリ（2026-02-25に手動仕訳テーブルから分離）
- `account_items` 分類: `small_category='繰延資産', large_category='資産'`
- `pl_contribution = NULL` → **P&L計算に一切含まれない**
- 2026年2月現在まで **一切償却せず** 繰延資産として保持中

### 4.2 償却開始の条件

開業費の償却は任意（任意償却）。開始時期・償却期間は事業者が選択可能。

償却開始時の処理:
```
Dr. 繰延資産償却費 / Cr. 開業費
```

→ 償却開始する場合は `繰延資産償却費` 科目を account_items に追加し、`small_category='経費'` に設定する。

## 5. 勘定科目分類方針

### 5.1 Amazon手数料の細分化

MFでは「Amazon手数料」1科目で一括計上していたが、BQでは費用種別ごとに細分化:

| BQ科目 | 内容 | MFでの対応 |
|---|---|---|
| 諸会費 | Amazonプロプラン月会費、各種サービス料 | Amazon手数料（一括） |
| 荷造運賃 | FBA手数料（配送・倉庫保管） | Amazon手数料（一括） |
| 販売手数料 | カテゴリ別紹介手数料 | Amazon手数料（一括） |

→ BQ方式の方が費用分析に有用。継続する。

### 5.2 外注費の範囲

BQでは代行会社への間接費（梱包・検品・ラベル等）を全て「外注費」に計上。
MFでは「外注工賃」+「研修採用費」に分散していた。
→ BQ方式を継続する。

## 6. 原価計算方針

### 6.1 基本方針

| 用途 | 原価の種類 | 元データ |
|---|---|---|
| BQ月次損益分析 | 標準原価 × 販売数量 | analytics.stg_unit_costs |
| BQ仕入高計上 | 実際支払額（直近入金レート換算） | nocodb.agency_transactions |
| 棚卸評価 | 標準原価（年1回手動更新） | nocodb.standard_cost_history |

- 為替レート: 振替テーブルに記録した直近入金レートを使用（税関公示レートは不使用）
- SKU別実勢原価按分は行わない（代行会社でのセット組みが介在するため精度不足）

### 6.2 標準原価の年次更新手順

1. 振替テーブルで年間の間接費率を算出（間接費率 = (振替合計 - 商品代金合計) / 商品代金合計）
2. 発注明細で SKU 別の加重平均仕入単価を算出（NocoDB内で手計算）
3. 標準原価_翌年 = 加重平均仕入単価 × (1 + 間接費率)
4. NocoDB「標準原価履歴」テーブルに登録 → BQへエクスポート

## 7. 棚卸資産の評価方針

### 7.1 評価方法

**標準原価法**を採用。SKU別の標準原価は年1回手動更新し、`nocodb.standard_cost_history` で管理。

### 7.2 棚卸高の算出

`accounting.inventory_journal_view` で自動生成:

- **在庫数量:** `sp_api_external.ledger-summary-view-data` の12月末 `Ending_Warehouse_Balance`（SELLABLE）
- **評価単価:** `nocodb.standard_cost_history` の当該年度標準原価
- **結合キー:** product_master.amazon_sku = ledger-summary-view-data.MSKU

### 7.3 仕訳パターン

```
期首: Dr. 仕入高（売上原価） / Cr. 商品（棚卸資産）
期末: Dr. 商品（棚卸資産） / Cr. 仕入高（売上原価）
```

### 7.4 原価の年度対応ルール

FY Y の棚卸仕訳（期首・期末とも）は FY Y の標準原価で評価:
- 期首（12/(Y-1)在庫） × FY Y 原価
- 期末（12/Y 在庫） × FY Y 原価

→ 同一年度内で一貫した原価評価を保証。年度間の原価変更はP&Lに反映される。

### 7.5 年度選定

期首・期末の両方のデータが存在する年度のみ仕訳を生成。
→ 2023/2024年度（MF照合済み）のP&Lを保護。

### 7.6 2025年度実績

| 項目 | 金額 | SKU数 |
|------|------|-------|
| 期首棚卸高（2024/12/31） | ¥483,968 | 13 SKU |
| 期末棚卸高（2025/12/31） | ¥502,320 | 11 SKU |
| P&L影響（仕入高純減） | +¥18,352 | |

## 8. 2025年以降の運用方針

### 8.1 MFとの照合は不要

2025年以降、マネーフォワードとの照合は行わない。BQの計算値が唯一の会計記録となる。

### 8.2 差異調整仕訳は不要

MF照合用の差異調整仕訳（Id=190,191,192のような残差・タイミング調整）は不要。
手動仕訳テーブルへの新規入力は以下のケースのみ:
- 事業主借経費（楽天カード等で記録されていない事業支出）
- 期末為替評価替え（上記 §3 参照）

### 8.3 freee連携時の仕訳日基準

deposit_date ベースを推奨（上記 §1 参照）。
