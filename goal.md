# 理想の状態

- amazon-sp-apiから各種データを自動取得できている
- amazon-ads-apiから広告データを取得できている
- 可能な限り網羅的にデータを取得することで、多角的なデータ分析ができる
- 取得した生データはGoogle Cloud Storage (GCS) に保存されている
- GCSに保存されたデータを使ってBigQueryを中心とした分析基盤が構築できている
- 自動取得が困難なデータのみGoogle Sheetsで手動管理し、BigQueryから参照できている
- 日々のデータ取得・保存が自動化されている
- 商品別分析には日次データと週次データの両方を活用できている
- 販売利益（売上 - 原価 - 広告費）をSKU別に把握できている
- 日々の販売利益をドリルダウンすることで日々の利益変動の要因を限界まで分解して詳細な分析ができている
  - 月間販売利益=商品別販売利益A+商品別販売利益B+商品別販売利益C-固定経費
  - 商品別販売利益 = 広告売上高 + 自然売上高 - 売上原価 - 販売手数料 - 広告費
    = 広告imp _ 広告CTR _ (広告CVR _ 販売単価 - 原価単価 - 販売手数料単価 - CPC) + 自然imp _ 自然CTR \* 自然CVR (販売単価 - 原価単価 - 販売手数料単価)
- さらに市場の影響と自社の実力値による数値変動を切り分けることができている（SP-APIのSQPや広告インプレッションシェアなど）
- 輸入代行会社から共有される発注・輸送コスト情報（不定形なGoogle Sheetsデータ）を最小限の工数でbigqueryの分析基盤に統合できている
- 輸入代行会社への直近入金レートを使用した適切な為替換算ができている
- 以下の課題をクリアして商品ごとの実勢原価を可能な限り正確に把握できている
  - 商品本体と包材などを別々に購入して代行会社でセット組みしている
  - セット商品をバラ売りしている場合がある
  - 代行会社で商品本体、包材、同梱物などを輸入前に一時在庫する場合がある
  - 中国工場は不良率が高く、返金ではなく返品対応になるため発注数量と実際の仕入数量が異なる場合がある
  - 前回返品数量を考慮した納品数量も若干ズレる場合がある
  - 発注数量は発注書、納品数量と代FBAへの出荷（輸入数量）はパッキングリストに不定形に記載されており、手動でデータベース化が必要。
  - 副業の個人事業主であるためデータ整形には限界がある
  - 一律に数量や金額で配賦すると、同じ輸入ロットに複数品目が混ざったときに影響の大きな送料や検品費用の配賦が不正確になる
  - 商品本体の仕入れ原価が変動する場合がある
  - 中国国内送料、国際送料、関税消費税等を正確にSKU別に配賦する必要がある
- freeeを中心に日々のbigquery分析基盤と紐づいた会計処理が自動化されている
- Amazon販売データの会計処理には銀行への送金と紐づけやすいsettlementレポートv2を使用している
- 代行会社経由で支払う仕入高についても可能な限り自動化されている
- 代行会社に預け金を入金して残高から支払う方式を採用している仕入高も会計上適切でかつ、分析基盤と紐づけやすい形で自動化されている
- 為替レートの取得と換算も自動化されている
- 月次のAmazon在庫レポートを活用した月次棚卸高、もしくは決済レポートに紐づいた売上原価の計上でが月次決算が自動化できている
- すでに確定申告済みの2023年度及び2024年度のデータについても遡及的に分析基盤に取り込み、freeeの会計データと突合できている
- 2025年移行と同様の分析基盤を構築すると共に、修正申告は避けたいため、法的に推奨される場合、2023年度と2024年度実績の申告数値とのずれは、なんらかの調整勘定で吸収できている
- 2025年度以降は、完全な自動化と高精度な商品別販売利益分析が実現できている
- トラフィックデータに紐づく日々の分析とは別に、settelementレポートv2に基づく注文ごとの利益分析もfreeeのその他経費と統合して集計分析できるようになっている
- 確定申告済みデータに誤りがある場合は、適切な形に修正したうえで、適切な調整仕訳で確定申告との数値ズレを補正できている
